# business-proposal Skill

接收一個客戶資料夾路徑，閱讀裡面的專案素材，指揮三位專家 Sub-Agent 協作，
提出**三種不同切入點**的企業形象影片提案企劃，產出**一份**對標範例格式的報告。

全程**全自動，中間不停下來等使用者回覆**，四個 Phase 走完後統一報告一次。

---

## 輸入

使用者提供：**一個客戶資料夾的路徑**（例：`/Users/masen/Documents/ap/客戶/麥當勞/`）。

資料夾內的約定結構：

| 項目 | 說明 | 用途 |
|------|------|------|
| `專案內容資料夾/` | 客戶提供的原始素材（簡報、需求文件、品牌資料、產品照、競品連結等，格式不限） | 提案的事實依據 |
| `影片企劃範例.*` | 一份既有的影片企劃文件（.docx / .pptx / .pdf 皆可） | 產出報告要對標的**格式範本** |

從**客戶資料夾的名稱**取得客戶名稱，用於報告命名與內文。

---

## Phase 1：盤點與閱讀

### Step 1.1：定位資料夾與範例

```bash
ls -R "<客戶資料夾路徑>"
```

找出 `專案內容資料夾`（素材）與檔名含「企劃範例 / 範例 / template / 範本」的檔案（格式範本）。
找不到對應檔案時不中斷——在最終報告標注，並用本 skill 的預設企劃結構（見 Phase 4）。

### Step 1.2：讀完所有專案素材

把 `專案內容資料夾` 裡每一份檔案都讀進來，不略讀：
- `.docx` 用 python-docx；`.pptx` 用 python-pptx；`.pdf` 用 Read 工具的 PDF 能力；`.txt/.md` 直接讀。
- 圖片用 Read 工具看內容（品牌調性、產品外觀、既有視覺）。
- 條列出**客觀事實**：客戶是誰、賣什麼、目標市場、競爭環境、品牌個性、明確的喜好或禁忌。這份事實清單會傳給三位 Sub-Agent，避免他們各自重讀。

### Step 1.3：解析格式範本，判斷它是「簡報」還是「文件」

讀 `影片企劃範例`，**先判斷它的本質**——這決定 Phase 4 的輸出格式：

| 範本長相 | 判定 | 輸出 |
|---------|------|------|
| 投影片式（.pptx，或 .pdf 但每頁是一張 16:9 版面、有區塊頁/主視覺頁） | **簡報** | `.pptx`（走 Phase 4，已有定案版型可直接套） |
| 連續文字/表格的 Word 文件 | **文件** | `.docx`（沿用文件版型，見 Phase 4 末段） |

> 實務經驗：綠芽的「影片企劃範例」通常是**簡報**（鼎康範例就是 19 頁 16:9 PDF）。預設往 `.pptx` 走，除非範本明顯是純文字文件。

萃取範本的視覺語言（區塊順序、主色、字型、單一提案頁怎麼長），記下來給 Phase 4。
注意：客戶資料夾裡可能還有 `提案腳本架構（範本）.docx` 之類的**分鏡腳本範本**——那是 script-writer 的範疇，不是本 skill 的對標對象，別搞混。

## Phase 2：三位專家協作（平行 Sub-Agent）

**在同一個回合內平行啟動三個 Sub-Agent**（用 Agent / Task 工具，subagent_type 用 `general-purpose`）。
這些角色尚未註冊成獨立 agent type，所以每個 Sub-Agent 第一步要先讀人設**進入角色**，再分析。

**進入角色的讀法（重要）**：每個 Sub-Agent 先讀本 skill 的精簡人設
`references/persona_<角色>.md`（已蒸餾、token 少、角色穩）；
**只有當該檔還是空模板（未填內容）時**，才退回讀完整知識庫。
所有路徑一律用**絕對路徑**——Sub-Agent 的工作目錄不保證在專案根，相對路徑會讀不到。

三個 Sub-Agent 都收到 Step 1.2 的事實清單，各自從專業視角產出**素材**（這階段不寫簡報，只貢獻觀點）：

### Sub-Agent A — 市場分析師（吳柏諺）
- 精簡人設：`/Users/masen/Documents/ap/.claude/skills/business-proposal/references/persona_市場分析師.md`
- 退回讀（模板未填時）：`/Users/masen/Documents/ap/Agent/市場分析師/soul.md` ＋ `…/knowledge/廣告專案市場分析知識庫.md`、`市場分析框架.md`
- 產出：目標受眾輪廓（Persona）、市場/競爭洞察、**2–3 個有市場依據的策略切入方向**（每個說清楚為什麼打得中）

### Sub-Agent B — 創意總監（張雅婷）
- 精簡人設：`/Users/masen/Documents/ap/.claude/skills/business-proposal/references/persona_創意總監.md`
- 退回讀：`/Users/masen/Documents/ap/Agent/創意總監/soul.md` ＋ `…/knowledge/影片廣告創意製作知識庫.md`、`創意總監核心能力知識庫.md`、`奧格威廣告哲學知識庫.md`
- 產出：**2–3 個 Big Idea**（核心概念 + Slogan（含英文）+ 情感主軸）、為什麼能讓人記住想分享

### Sub-Agent C — 導演（Sebastian）
- 精簡人設：`/Users/masen/Documents/ap/.claude/skills/business-proposal/references/persona_導演.md`
- 退回讀：`/Users/masen/Documents/ap/Agent/導演/soul.md` ＋ `…/knowledge/廣告影片製作知識庫.md`、`導演核心能力.md`
- 產出：**2–3 個影像處理方向（Treatment）**（視覺風格、色調、鏡頭語言、節奏、開場與收尾畫面）、每個一句話畫面感（供 deck 的 `image_hint`/`film`）

三個 Sub-Agent 完成後，把它們的產出收回主線。

---

## Phase 3：整合成三種切入點

主線把三位專家的素材**交叉組合**成**三個完整、彼此明顯不同**的提案切入點。
關鍵：三案不是「市場案 / 創意案 / 導演案」，而是三個都由三種視角共同支撐的完整提案——
每一案都同時回答「打中誰、用什麼大創意、長什麼樣子」。

每個切入點包含：

| 欄位 | 來源 | 內容 |
|------|------|------|
| 切入點名稱 | 主線命名 | 好記、能一句話定調，例：「以人為本」「技術即信任」 |
| 一句話定位 | 創意總監 | 這一案的核心主張 |
| 市場依據 | 市場分析師 | 為什麼這角度打得中目標受眾/市場 |
| 核心創意 Big Idea | 創意總監 | 核心概念 + Slogan + 情感主軸 |
| 影像方向 Treatment | 導演 | 視覺風格、色調、鏡頭語言、節奏、代表性畫面 |
| 適合理由 | 主線 | 一句話：什麼情況下客戶該選這一案 |

三案要刻意拉開差異（保守 / 大膽、理性 / 感性、產品導向 / 品牌精神導向等），給客戶真正的選擇。

## Phase 4：產出簡報 .pptx（定案格式，對標鼎康範例）

**已定案版型**：直接用本 skill 內建的產生器，不要重寫繪圖程式碼。

```python
import sys
sys.path.insert(0, "/Users/masen/Documents/ap/.claude/skills/business-proposal/scripts")
from proposal_deck import build_deck

build_deck(
    client_name="【客戶名稱】",
    angles=[ angle1, angle2, angle3 ],   # 每個 angle 的欄位見下方
    out_path="<客戶資料夾路徑>/【客戶名稱】企業形象提案企劃.pptx",
    context=context,                     # 選填：市場洞察頁，見下方
)
```

**你的工作只是把 Phase 3 的三案內容填進 `angles`**。每個 angle 是一個 dict：

| 欄位 | 內容 | 來源 |
|------|------|------|
| `no` | `"01"` / `"02"` / `"03"` | — |
| `name` | 切入點名稱（如「可被驗證的信任」） | Phase 3 |
| `hero_zh` | 主視覺中文大標，用 `\n` 斷成 1–2 行 | 創意總監 |
| `slogan` | 英文 Slogan（broadcast 等級） | 創意總監 |
| `sub` | 英文副標一句 | 創意總監 |
| `statement` | 中文核心陳述一句 | 創意總監 |
| `image_hint` | 主視覺示意一句（這頁該放什麼畫面，標給設計/後製） | 導演 |
| `market` | **市場依據一句**（為什麼這角度打得中買家） | 市場分析師 |
| `concepts` | **3 個** `(中文概念, 英文, 一句說明)` → 影片架構概念圈 | 三方整合 |
| `labels` | 架構底線標籤 2 個，如 `["Traceable","Reliable"]` | 主線 |
| `outline` | 內容大綱段落 | 主線 |
| `film` | 影像方向一句話 | 導演 |
| `fit` | 「適合：…」一句話 | 主線 |

`context`（選填，建議都給，讓市場分析師的工被看見）：
```python
context = dict(
    summary="客戶一句話：是誰、賣什麼、主打什麼",
    audience="目標受眾",
    insights=["市場洞察 1", "市場洞察 2", "市場洞察 3"],   # 3 點
)
```

完整填寫範例見 `scripts/proposal_deck.py` 末段的品香茶葉 `__main__`（定案內容，可當新案範本）。

**版型固定**（`proposal_deck.py` 已實作，勿改）：封面 → 簡報大綱 → 專案理解·市場洞察（給了 context 才出）→ 每案 3 頁（區塊頁 / 核心訊息主視覺〔含主視覺示意〕/ 影片架構＋提案重點〔市場依據·內容大綱·影像方向·適合〕）→ THANK YOU。給 context 時共 13 頁。
視覺系統：深藍 `#0C2340`、主藍 `#1565C0`、淺藍 `#DBEAF7`、品牌綠 `#2ED15A`（僅封面/結尾）；16:9；**照片區一律用原生色塊，禁用 JPG/PNG**。

### Step 4.1：版面驗證（必跑，不可跳過）

```bash
python3 /Users/masen/Documents/ap/.claude/skills/pptx-builder/scripts/validate_pptx.py "<輸出路徑>"
```

要看到 `✓ 全部 N 頁：無 overflow，無文字重疊`。若內容太長導致 overflow，**精簡文字**（concepts 說明、outline）後重跑，最多 3 次。

---

### （備案）範本是純文字 Word 文件時 → 輸出 .docx

只有在 Phase 1.3 判定範本是**純文字文件**時才走這條：用 python-docx 產出一份 `.docx`，標題 14pt bold、主色沿用範本，三案依序排版（定位 / 市場依據 / Big Idea+Slogan / 影像 Treatment / 適合理由）。
表格欄寬務必先移除自動 `w:tcW` 再插入（否則 Word 只讀第一個等寬值）。

## 錯誤處理（不中斷流程）

| 問題 | 處理方式 |
|------|---------|
| 客戶資料夾路徑找不到 | 立即停下，請使用者確認路徑（這是唯一會停的情況） |
| 找不到「專案內容資料夾」 | 改讀客戶資料夾根目錄的所有素材，並在報告標注 |
| 找不到「影片企劃範例」 | 用 Phase 4 的預設結構，並在報告標注「⚠️ 未找到範例」 |
| Sub-Agent 某個失敗 | 主線用該角色知識庫自行補上該視角，報告標注 |
| python-pptx 未安裝 | 立即 `pip install python-pptx`（備案 docx 則裝 python-docx）後繼續 |
| 驗證 3 次仍 overflow | 在最終報告列出殘留頁，並建議精簡方向 |
| 存檔目錄不存在 | 自動 `mkdir -p` 建立 |
